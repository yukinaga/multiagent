────────────────────────────────────────
4.1 サブエージェントとは何か
────────────────────────────────────────

■ サブエージェントの概念
  メインのClaude Code ＝ プロジェクトマネージャー（PM）
  サブエージェント     ＝ 専門家チームのメンバー

■ コンテキスト分離の3つのメリット
  ① メインの会話が汚れない（サマリーだけが返る）
  ② 専門的な指示をシステムプロンプトに集約できる
  ③ コンテキストウィンドウの消費が最小限で済む

■ サブエージェントの種類
  【組み込み】Claudeが自動で使い分ける
    Explore          読み取り専用。コードベースの検索・分析
    Plan             Plan mode時に使用。調査して計画情報を収集
    General-purpose  探索＋実行の両方が可能。複雑なタスク向け

  【カスタム】自分で作る専門家
    .claude/agents/ にMarkdownファイルを配置する


────────────────────────────────────────
4.2 組み込みサブエージェントの動きを観察する
────────────────────────────────────────

■ 会話をリセット
  /clear

■ Exploreサブエージェントの自動委任を観察するプロンプト
  このプロジェクトの構造と各ファイルの役割を詳しくまとめてください。

■ サブエージェントが使われない場合（簡単な質問）との比較
  src/todo.pyの3行目には何が書いてありますか？

■ サブエージェント呼び出しの目印（VSCode上での表示）
  ● Task(プロジェクト構造の調査)
  ⎿ Done (17 tool uses · 56.6k tokens · 1m 34.3s)

  → ツール使用回数、トークン消費量、所要時間がわかる

■ サブエージェントが使われる基準
  - タスクが複雑で広範囲の探索が必要なとき
  - メインのコンテキストを汚したくないとき
  - 単純な質問にはClaude本体が直接対応する


────────────────────────────────────────
4.3 カスタムサブエージェントを作る①：コードレビュアー
────────────────────────────────────────

【方法A：/agents コマンドで作成】

■ エージェント管理メニューを開く
  /agents

■ 作成フロー
  1. 「Create new agent」を選択
  2. スコープを選択：「Project-level」（プロジェクト専用）
     ※ User-level を選ぶと ~/.claude/agents/ に保存（全プロジェクト共通）
  3. 作成方法を選択：「Generate with Claude」
  4. 説明を入力：
     コードレビューの専門家。バグ、セキュリティリスク、パフォーマンスの観点で
     Pythonコードをレビューする。変更は加えず、問題点の指摘と改善提案だけを行う。

■ 生成されるファイル
  .claude/agents/code-reviewer.md

■ code-reviewer.md の内容
─────────────────────────────
---
name: code-reviewer
description: コードの品質をレビューし、バグ、セキュリティリスク、
  パフォーマンスの問題を指摘する専門家エージェント
tools: Read, Grep, Glob
model: sonnet
---

あなたはシニアソフトウェアエンジニアとして、
コードレビューを行う専門家です。

## レビュー観点
以下の観点でコードを分析してください：

1. **バグ・論理エラー**
   - 境界値の処理漏れ
   - 例外処理の不足
   - 型の不整合

2. **セキュリティリスク**
   - 入力値の未検証
   - 機密情報のハードコーディング
   - インジェクションの可能性

3. **パフォーマンス**
   - 不要なループや重複処理
   - メモリ効率の悪いデータ構造
   - N+1問題

## 出力フォーマット
レビュー結果は以下の形式で報告してください：
- 重要度（高/中/低）
- 該当箇所（ファイル名と行番号）
- 問題の説明
- 改善案
─────────────────────────────

■ フロントマターの各項目
  name         呼び出し名（「code-reviewerを使って」で指名可能）
  description  自動委任の判断基準 ⭐最重要
  tools        使えるツールの制限（Read, Grep, Glob = 読み取り専用）
  model        使用モデル（sonnet / opus / haiku）

■ code-reviewerを使ってレビューするプロンプト
  /clear
  code-reviewerサブエージェントを使って、src/todo.py をレビューしてください。


────────────────────────────────────────
4.4 カスタムサブエージェントを作る②：テストエンジニア
────────────────────────────────────────

【方法B：Markdownファイルを手動作成】

■ ファイルを手動で作成
  .claude/agents/test-engineer.md

■ test-engineer.md の内容
─────────────────────────────
---
name: test-engineer
description: テストコードの作成と実行を担当する専門家エージェント。
  テストカバレッジの改善とテスト品質の向上を行う。
tools: Read, Write, Edit, Bash, Glob, Grep
model: sonnet
---

あなたはテスト自動化の専門家です。

## 基本ルール
- テストフレームワークはpytestを使用する
- テスト実行コマンドは `uv run pytest` を使用する（pipやpython -m pytestは使わない）
- テストファイルは tests/ ディレクトリに配置する
- ファイル名は test_対象モジュール名.py とする

## テスト方針
- 正常系と異常系の両方をカバーする
- 境界値テストを含める
- テストは独立して実行可能であること（他のテストに依存しない）
- 各テスト関数には日本語のdocstringで何をテストしているか書く

## 作業手順
1. 対象コードを読んで、テストすべき機能を洗い出す
2. テストコードを作成する
3. `uv run pytest` でテストを実行する
4. 失敗がある場合はテストコードを修正して再実行する
5. 全テストがパスしたらサマリーを報告する

## 出力フォーマット
テスト実行後、以下のサマリーを必ず報告する：
1. テスト件数（パス / 失敗 / スキップ）
2. カバレッジが低い箇所の指摘
3. 追加すべきテストケースの提案
─────────────────────────────

■ 事前準備：pytestのインストール
  uv add --dev pytest

■ test-engineerを使ってテストを作成するプロンプト
  /clear
  test-engineerサブエージェントを使って、src/todo.py の
  Todoクラスに対するテストを作成してください。
  正常系と異常系をそれぞれ3ケース以上含めてください。


================================================================
2つのエージェントの比較
================================================================

  ┌──────────────┬──────────────────────────┬──────────────────────────────────┐
  │              │ code-reviewer            │ test-engineer                    │
  ├──────────────┼──────────────────────────┼──────────────────────────────────┤
  │ 役割         │ コードの問題点を指摘     │ テストを書いて実行する           │
  │ ツール       │ Read, Grep, Glob         │ Read, Write, Edit, Bash,         │
  │              │                          │ Glob, Grep                       │
  │ できること   │ ファイルを読むだけ       │ ファイルの読み書き＋コマンド実行 │
  │ 安全性       │ コードを変更できない     │ テストファイルを作成・変更できる │
  │ 出力         │ レビューレポート         │ テストコード＋実行結果           │
  └──────────────┴──────────────────────────┴──────────────────────────────────┘

  設計原則：エージェントに許可するツールは、役割に必要な最小限にする


================================================================
エージェントファイルの構造
================================================================

  .claude/agents/エージェント名.md
  ├── フロントマター（YAML、--- で囲む）
  │   ├── name         → 呼び出し名
  │   ├── description  → 自動委任の判断基準 ⭐最重要
  │   ├── tools        → 使えるツールの制限
  │   └── model        → 使用モデル
  └── 本文（Markdown）
      └── システムプロンプト → 振る舞いの定義


================================================================
descriptionの書き方のコツ
================================================================

  ■ 悪い例
    テストを担当するエージェント

  ■ 良い例
    テストコードの作成と実行を担当する専門家エージェント。
    テストカバレッジの改善とテスト品質の向上を行う。

  ■ チェックポイント
    ① 「いつ使われるべきか」がdescriptionから読み取れるか？
    ② 他のエージェントと区別できるか？
    ③ 短すぎず長すぎないか？（1〜2文が目安）
