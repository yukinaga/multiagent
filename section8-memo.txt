────────────────────────────────────────
8.1 いつ何を使うか：判断フローチャート
────────────────────────────────────────

■ 判断フローチャート（4つの質問）

  タスクを受け取った
      │
      ▼
  ① 複雑さはどのくらい？
      ├─ シンプル（1ファイル、1機能）
      │   → 通常のClaude Code ✅
      │
      └─ 複雑（複数の関心事がある）
          │
          ▼
  ② コンテキストを分けたい？
      ├─ No → 通常のClaude Code + Plan mode ✅
      │
      └─ Yes
          │
          ▼
  ③ エージェント同士の連携が必要？
      ├─ No → サブエージェント ✅
      │
      └─ Yes
          │
          ▼
  ④ 並列の価値 > コスト？
      ├─ No → サブエージェントを順次実行 ✅
      └─ Yes → Agent Teams ✅


■ 4つの判断キーワード
  ① 複雑さ       → シンプルなら通常で十分
  ② 分離の必要性 → コンテキストが汚れるなら分ける
  ③ 連携の有無   → 独立なら順次、連携ならチーム
  ④ コスト対効果  → 並列の時間短縮に価値があるか


■ シナリオ別の判断例

  シナリオ①：1ファイルのバグ修正
    「login.py の認証エラーを直して」
    → ① シンプル → 通常のClaude Code

  シナリオ②：コードレビュー
    「src/ のコード品質をチェックして」
    → ① 複雑 → ② 分離Yes → ③ 連携No → サブエージェント（code-reviewer）

  シナリオ③：新機能追加（1モジュール）
    「ユーザープロフィール機能を追加して」
    → ① 複雑 → ② 分離No（一貫性を保ちたい） → 通常 + Plan mode

  シナリオ④：品質改善パイプライン
    「レビュー → テスト追加 → ドキュメント生成」
    → ① 複雑 → ② 分離Yes → ③ 連携No → サブエージェントを順次実行

  シナリオ⑤：REST APIの並列開発
    「API / テスト / ドキュメントを同時に作って」
    → ① 複雑 → ② 分離Yes → ③ 連携Yes → ④ コスト見合う → Agent Teams


■ よくある間違い
  ❌「複雑だからAgent Teams」
    → 分離が不要なら通常 + Plan mode で十分
  ❌「サブエージェントがあるから何でも委任」
    → シンプルなタスクはオーバーヘッドで逆に遅くなる
  ❌「Agent Teamsは常にサブエージェントより上位互換」
    → 連携不要なら順次実行のほうが安定・安い


■ グレーゾーンの原則
  ① 迷ったらシンプルな方を選ぶ
    通常 → サブエージェント → Agent Teams
    スケールアップは簡単、スケールダウンは面倒
  ② やってみて判断でOK
    通常で始めて、コンテキストが長くなったらサブエージェントに切り替え
  ③ 完璧な判断より、素早い判断
    どれを使っても結果は出る。選択に悩む時間が一番もったいない


────────────────────────────────────────
8.2 コスト管理の実践
────────────────────────────────────────

■ トークンの基本
  1トークン ≈ 日本語1文字 / 英語4文字
  入力トークン（安い）：あなたが送る指示
  出力トークン（高い）：Claudeが返す応答（入力の約3〜5倍）
  エージェントループが長いほどコストが増える

■ 3つの手段のコスト感覚
  ① 通常のClaude Code      基準（1倍）
  ② サブエージェント        約1.5〜2倍
  ③ Agent Teams（3人）      約3倍


■ コスト最適化テクニック（効果の大きい順）

  テクニック①：/clear をこまめに使う（効果：最大）
    タスク完了 → /clear → 新しいタスク
    コンテキスト肥大を防ぎ、入力トークン増加を抑える

  テクニック②：Plan mode で先に計画する（効果：大）
    Shift+Tab でPlan mode → 計画を承認 → Normal で実行
    手戻りを減らし、無駄なWrite/Bashを防ぐ

  テクニック③：サブエージェントのモデルを変更する（効果：中）
    .claude/agents/ のフロントマターで指定：
      ---
      model: claude-sonnet-4-6
      ---
    メインはOpus、サブはSonnetで使い分け

  テクニック④：Agent Teams は本当に必要なときだけ（効果：大）
    並列で得られる時間短縮 > メンバー数倍のコスト → Agent Teams
    そうでなければ → サブエージェントを順次実行

  📌 まず①と②を習慣にするだけでコストは大きく変わる


■ 料金プランの選び方
  ┌─────────────────────────────────────────────────────────┐
  │ 学習目的・たまに使う                                   │
  │ → Claude Pro（月額 $20）                               │
  │   このコースの演習はこれで十分                         │
  ├─────────────────────────────────────────────────────────┤
  │ 日常的にClaude Codeで開発する                          │
  │ → Claude Max 5x（月額 $100）                           │
  │   Proの5倍の使用量。Agent Teamsも実用的に使える       │
  ├─────────────────────────────────────────────────────────┤
  │ ヘビーユーザー・大規模プロジェクト                     │
  │ → Claude Max 20x（月額 $200）                          │
  │   Proの20倍。ほぼ制限を気にせず使える                 │
  └─────────────────────────────────────────────────────────┘
  ※ 最新の料金は claude.ai/pricing で確認

■ API利用の場合（参考）
  Sonnet：入力 $3 / 出力 $15（100万トークンあたり）
  Opus ：入力 $15 / 出力 $75（100万トークンあたり）
  ※ spending limit（月額上限）の設定を推奨

■ セッション中のコスト確認コマンド
  /cost

  表示される情報：
    Input tokens:  入力トークン数
    Output tokens: 出力トークン数
    Estimated cost: API換算の推定コスト
  ※ サブスクリプションの場合は参考値

■ コスト感覚の目安
  簡単なバグ修正         約 $0.1
  大きなリファクタリング 約 $1
  Agent Teams 並列開発    約 $3〜5


────────────────────────────────────────
8.3 チームの設計パターン
────────────────────────────────────────

■ 4つの設計パターン一覧
  ┌──────────────────┬──────────────────┬────────────────────┐
  │ パターン         │ キーワード       │ 手段               │
  ├──────────────────┼──────────────────┼────────────────────┤
  │ ① レイヤー分担型 │「一緒に作る」    │ Agent Teams        │
  │ ② 並列リサーチ型 │「比べて選ぶ」    │ Agent Teams        │
  │ ③ 競争的仮説型   │「原因を探る」    │ Agent Teams        │
  │ ④ パイプライン型 │「順番に磨く」    │ サブエージェント   │
  └──────────────────┴──────────────────┴────────────────────┘

  ①〜③ は Agent Teams 向き
  ④ は サブエージェント向き（並列にできない → Agent Teamsのメリットなし）


■ パターン①：レイヤー分担型（Agent Teams）
  各メンバーが異なるレイヤー（層）を担当する

  向いている場面：
    新しい機能やアプリケーションの構築
    各メンバーが異なるファイルを担当できるとき

  構成例：
    メンバー1：APIエンドポイント担当 → src/main.py, src/models.py, src/database.py
    メンバー2：テスト担当           → tests/test_api.py
    メンバー3：ドキュメント担当     → README.md, docs/api-guide.md

  コツ：担当ファイルを明示してファイル衝突を防ぐ
  → セクション7のREST API開発がこのパターン


■ パターン②：並列リサーチ型（Agent Teams）
  複数の選択肢を同時に調査・比較する

  向いている場面：
    フレームワークやライブラリの選定
    技術的アプローチの比較検討
    リサーチや情報収集

  構成例：
    メンバー1：FastAPI を調査
    メンバー2：Litestar を調査
    メンバー3：Starlette を調査
    リーダー：比較表を作成し推薦をまとめる

  コツ：調査観点を揃えて比較しやすくする
  💡 コードを書かないのでリスクが低い。Agent Teams入門におすすめ
  → セクション6.3のリサーチチームがこのパターン


■ パターン③：競争的仮説型（Agent Teams）
  バグや問題の原因仮説を並列に検証する

  向いている場面：
    原因不明のバグ調査
    パフォーマンス問題の切り分け
    複数の解決策の実験的比較

  構成例：
    メンバー1：「DBクエリが原因」と仮定 → クエリの実行計画を分析
    メンバー2：「N+1問題が原因」と仮定 → ORMのクエリログを分析
    メンバー3：「外部API呼び出しが原因」と仮定 → レスポンスタイムを計測

  コツ：「仮説」と「検証方法」をセットで明示する
  📌 1人で順番に試すより圧倒的に速い


■ パターン④：パイプライン型（サブエージェント）
  前のステップの結果を次のステップに引き継ぐ

  向いている場面：
    品質改善ワークフロー
    段階的なデータ処理
    依存関係が明確な順次タスク

  構成例：
    [分析] → [実装] → [テスト] → [レビュー]

  ⚠️ Agent TeamsではなくサブエージェントGが適切
  理由：各ステップが前のステップに依存 → 並列にできない → Agent Teamsのメリットなし
  → セクション5の品質改善パイプラインがこのパターン


■ パターン選択の判断ヒント
  並列にできる？
    Yes → ①②③のどれか
    No  → ④（パイプライン型 = サブエージェント）

  並列にできる場合：
    コードを書く     → ① レイヤー分担型
    コードを書かない → ② 並列リサーチ型
    仮説を検証する   → ③ 競争的仮説型

よく使うコマンドまとめ（セクション8関連）
================================================================

  /model          使用モデルの表示・切り替え
  Shift+Tab       Plan modeに切り替え（コスト最適化テクニック②）
