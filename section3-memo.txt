────────────────────────────────────────
3.1 エージェントループを体験する
────────────────────────────────────────

■ 会話をリセットしてから始める
  /clear

■ 最初のタスク依頼（Claude Codeパネルに入力）
  src/todo.py に、Todoリストを管理するクラスを作成してください。
  追加、一覧表示、完了マーク、削除の機能を持つものにしてください

■ 生成されたコードの動作テスト
  uv run python -c "
  from src.todo import TodoList
  todos = TodoList()
  todos.add('牛乳を買う')
  todos.add('レポートを書く')
  print(todos.list_all())
  todos.complete(1)
  print(todos.list_all())
  "

  ※ Claudeが生成するクラス名・メソッド名は毎回異なる可能性あり。
    生成されたコードに合わせてコマンドを調整する。

■ importエラーが出た場合の対処
  touch src/__init__.py

■ エージェントループの観察ポイント
  ① Claudeはまず何をするか？（Read/Globでプロジェクト構造を確認）
  ② どんなツールを使うか？（Write, Edit, Bash など）
  ③ 何回ループするか？（計画→実行→確認→繰り返し）


────────────────────────────────────────
3.2 ツールとパーミッションを理解する
────────────────────────────────────────

■ ツール一覧（3カテゴリ）

  【読む系】（許可不要・自動実行）
    Read    ファイルの中身を読む
    Glob    パターンでファイルを探す
    Grep    テキストの中身を検索する

  【書く系】（許可が必要）
    Write   新しいファイルを作成する
    Edit    既存のファイルを編集する

  【実行系】（許可が必要）
    Bash      コマンドを実行する
    Task      サブエージェントを起動する
    WebFetch  Webページを取得する

■ パーミッションモードの切り替え
  Shift+Tab（繰り返し押して巡回）

  Normal       → 表示なし（デフォルト）
  Auto-accept  → 「auto-accept edit on」と表示
  Plan         → 「plan mode on」と表示

■ 3つのモードの比較
  ┌─────────────────┬──────────┬───────────┐
  │                 │ 読む系   │ 書く/実行  │
  ├─────────────────┼──────────┼───────────┤
  │ Normal          │ ✅ 自動  │ ⚠️ 確認    │
  │ Auto-accept     │ ✅ 自動  │ ✅ 自動    │
  │ Plan            │ ✅ 自動  │ 🚫 禁止    │
  └─────────────────┴──────────┴───────────┘

■ Normalモードでのデモ用プロンプト
  src/todo.py に、__str__ メソッドを追加してください。TodoListの中身を文字列で返すようにしてください

■ Auto-acceptモードでのデモ用プロンプト
  src/todo.py の __str__ メソッドに、完了済みタスクには ✓ マークを付ける処理を追加してください


────────────────────────────────────────
3.3 Plan modeで安全に計画を立てる
────────────────────────────────────────

■ Plan modeのワークフロー（4ステップ）
  ① Shift+Tab で「plan mode on」に切り替える
  ② 計画を依頼する
  ③ 計画を確認する（修正したければ追加指示を出す）
  ④ Shift+Tab でNormalに戻し「実行して」と伝える

■ Plan modeの使いどころ
  ① 大きな変更の前（複数ファイルにまたがる変更など）
  ② 方針を相談したいとき（最適な実装方法がわからないとき）
  ③ トークンを節約したいとき（やり直しを防ぐ）

■ Plan modeで計画を立てるプロンプト
  Todoアプリにカテゴリ機能を追加する計画を立ててください。
  各タスクに「仕事」「プライベート」などのカテゴリを設定できるようにしたいです。

■ Plan mode内で計画を修正するプロンプト
  いい計画ですね。ここに追加で、各タスクに優先度（高・中・低）も設定できるようにしたいのですが、計画を修正してもらえますか？

■ Normalに戻して実行を依頼するプロンプト
  この計画で実行してください


────────────────────────────────────────
3.4 CLAUDE.mdの威力を体験する
────────────────────────────────────────

■ CLAUDE.mdの階層構造（3つのレベル）
  ① プロジェクトルート   my-project/CLAUDE.md        → 全体に適用
  ② サブディレクトリ     my-project/tests/CLAUDE.md  → 追加で適用
  ③ ユーザーレベル       ~/.claude/CLAUDE.md          → 全プロジェクト共通

■ デモ1：CLAUDE.mdなしの状態を作る
  mv CLAUDE.md CLAUDE.md.bak

■ デモ1：CLAUDE.mdなしでパッケージ追加を依頼
  claude
  requests パッケージをインストールしてください
  → pip install が使われてしまう

■ デモ1：Claude Codeを終了
  /exit

■ デモ2：CLAUDE.mdを元に戻す
  mv CLAUDE.md.bak CLAUDE.md

■ デモ2：CLAUDE.mdありでパッケージ追加を依頼
  claude
  httpx パッケージを追加してください
  → uv add が使われる

■ デモ3：サブディレクトリCLAUDE.mdの作成プロンプト
  tests/CLAUDE.md を作成してください。内容は以下にしてください：

  ## テスト固有ルール
  - テストフレームワークはpytestを使用する
  - テストファイル名は test_対象モジュール名.py とする
  - 正常系と異常系の両方をカバーする
  - テストは独立して実行可能であること

■ デモ3：サブディレクトリのルールが効いているか確認
  tests/ ディレクトリに、todo.py のテストファイルを作成してください

■ 撮影後のクリーンアップ
  pip uninstall requests -y
  rm -f CLAUDE.md.bak pyproject.toml.bak
  git add .
  git commit -m "CLAUDE.mdデモ：tests/CLAUDE.mdを追加"


────────────────────────────────────────
3.5 効率的な使い方のコツ
────────────────────────────────────────

【コツ①：/clear の重要性】

■ /clearせずに別タスクを依頼（精度低下のデモ）
  src/todo.py に、期限（due_date）フィールドを追加してください

  （/clearせずにそのまま）

  新しいファイル src/calculator.py を作成して、四則演算ができるCalculatorクラスを作ってください

■ /clearしてから同じタスクを依頼（精度向上のデモ）
  /clear
  新しいファイル src/calculator.py を作成して、四則演算ができるCalculatorクラスを作ってください


【コツ②：良い指示の出し方】

■ 悪い例（曖昧な指示）
  テストを書いて

■ 良い例（具体的な指示）
  tests/test_todo.py に、Todoクラスの全メソッドに対するpytestテストを追加して。正常系3ケース、異常系2ケースを含め、実行して全パスすることを確認して

■ 良い指示の5つの要素
  ① どこに      → tests/test_todo.py
  ② 何を        → Todoクラスの全メソッド
  ③ どのように  → pytest
  ④ どれだけ    → 正常系3ケース、異常系2ケース
  ⑤ 最後にどう  → 実行して全パス確認


【コツ③：カスタムスラッシュコマンド】

■ コマンド用ディレクトリの作成
  mkdir -p .claude/commands

■ テストコマンドの作成
  ファイル: .claude/commands/test.md
  内容:
  $ARGUMENTS のテストを書いて実行して。正常系3ケース、異常系2ケースを含め、全パスすることを確認して

■ レビューコマンドの作成
  ファイル: .claude/commands/review.md
  内容:
  $ARGUMENTS をコードレビューして。バグ、パフォーマンスの問題、可読性の改善点を指摘して。修正案も提示して

■ カスタムコマンドの使用例
  /test src/todo.py
  /review src/todo.py

■ $ARGUMENTS の仕組み
  コマンド実行時に渡した引数に自動で置き換わる
  例: /test src/todo.py → $ARGUMENTS が src/todo.py に置換


【コツ④：モデルの切り替え】

■ モデル一覧を表示する
  /model

■ モデルの使い分け目安
  Opus    高精度。複雑なロジック、大きなリファクタリング、設計判断
  Sonnet  バランス型（デフォルト推奨）。日常的なコーディング全般
  Haiku   高速・低コスト。簡単な質問、ファイル検索、軽い修正


================================================================
セクション3 よく使うコマンドまとめ
================================================================

  /clear          会話をリセット（新しいタスクの前に毎回実行）
  /model          使用モデルの表示・切り替え
  /exit           Claude Codeの終了（ターミナル版）
  /test ...       カスタムコマンド（自分で作成）
  /review ...     カスタムコマンド（自分で作成）
  Shift+Tab       パーミッションモードの切り替え
                  Normal → Auto-accept → Plan → Normal
