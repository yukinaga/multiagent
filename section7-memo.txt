────────────────────────────────────────
7.1 プロジェクト概要と準備
────────────────────────────────────────

■ 構築するもの
  TodoアプリのREST API（CRUD操作）
    GET    /todos       一覧取得
    POST   /todos       新規作成
    PUT    /todos/{id}  更新
    DELETE /todos/{id}  削除

■ 技術スタック
  FastAPI    Webフレームワーク（APIのルーティング）
  SQLModel   ORM（データベースとPythonオブジェクトの変換）
  SQLite     データベース（ファイル1つで動く、セットアップ不要）
  httpx      非同期HTTPクライアント（APIテスト用）
  uvicorn    ASGIサーバー（FastAPIを動かすエンジン）

■ 依存パッケージの追加
  cd ~/multi-agent-course
  uv run python --version
  uv add fastapi uvicorn sqlmodel httpx

■ 開発用パッケージの追加
  uv add --dev pytest pytest-asyncio

■ pyproject.toml の確認
  cat pyproject.toml

■ CLAUDE.md に追記する内容
─────────────────────────────
## API開発ルール
- WebフレームワークはFastAPIを使用する
- ORMはSQLModelを使用する
- データベースはSQLite（開発環境）
- APIテストにはhttpxのAsyncClientを使用する
- サーバー起動コマンド: uv run uvicorn src.main:app --reload
- テスト実行コマンド: uv run pytest -v
─────────────────────────────

■ チーム構成（予告）
  チームメイト1：APIエンドポイント担当 → src/main.py, src/models.py, src/database.py
  チームメイト2：テスト担当           → tests/conftest.py, tests/test_api.py
  チームメイト3：ドキュメント担当     → README.md, docs/api-guide.md

  ※ 各メンバーの担当ファイルが重ならないように設計する（衝突防止）


────────────────────────────────────────
7.2 Agent Teamsで並列開発
────────────────────────────────────────

■ コンテキストをクリア
  /clear

■ Agent Teamsへの指示文（全文）
─────────────────────────────
TodoアプリのREST APIを構築します。
エージェントチームを作成して、以下の3つの担当に分かれて並列に作業してください。

チームメイト1：「APIエンドポイント担当」
- src/models.py にSQLModelのデータモデル（Todo）を作成
- src/database.py にSQLiteのデータベース接続・初期化処理を作成
- src/main.py にFastAPIアプリとCRUDエンドポイントを作成
  - GET /todos：一覧取得
  - POST /todos：新規作成
  - PUT /todos/{id}：更新
  - DELETE /todos/{id}：削除

チームメイト2：「テスト担当」
- tests/conftest.py にテスト用のフィクスチャを作成
- tests/test_api.py にAPIの統合テストを作成
  - httpxのAsyncClientを使った非同期テスト
  - 全エンドポイントの正常系テスト（各1ケース以上）
  - 異常系テスト（存在しないIDへのアクセスなど）
  - テスト実行: uv run pytest -v

チームメイト3：「ドキュメント担当」
- README.md をAPI仕様に合わせて更新
  - プロジェクト概要、セットアップ手順、エンドポイント一覧
- docs/api-guide.md にAPI使用ガイドを作成
  - 各エンドポイントのリクエスト・レスポンス例を含める

ルール：
- 各担当は指定されたファイルのみを編集すること（他の担当のファイルは触らない）
- API担当がモデルとエンドポイントを完成したら、テスト担当とドキュメント担当に通知すること
- テスト担当はテストが全パスしてから完了とすること
- チームメイトに作業を委任し、リーダー自身はコードを書かないこと

Agent Teams、tmuxを使い、別々のペインで各チームメイトを動作させてください。
─────────────────────────────

■ 良い指示文の3要素
  ① チーム構成：誰が何人か
  ② タスク定義：各メンバーの具体的な作業内容とファイル名
  ③ 制約・ルール：ファイル分離・通知タイミング・完了条件

■ 観察チェックリスト
  □ チーム作成 → tmuxペインが分割されたか？
  □ 各メンバー → 正しいファイルを担当しているか？
  □ メッセージング → API担当からテスト担当への通知があったか？
  □ タスク依存 → テスト担当がAPI完成を待つ場面があったか？
  □ 統合 → リーダーが結果をまとめたか？

■ 完了後のファイル確認
  ls -la src/
  ls -la tests/
  ls -la docs/

7.3 結果の確認とトラブルシューティング
────────────────────────────────────────

【ステップ1：ファイル構成の確認】

■ 生成されたファイルの一覧

■ 期待するファイル
  src/models.py       データモデル

  tests/conftest.py   テストのフィクスチャ
  tests/test_api.py   APIテスト
  docs/api-guide.md   API使用ガイド
  README.md           プロジェクトREADME（更新済み）


【ステップ2：APIサーバー起動 → Swagger UIで操作】

■ サーバー起動
  uv run uvicorn src.main:app --reload

■ Swagger UIにアクセス（ブラウザで開く）
  http://localhost:8000/docs

■ Swagger UIでのテスト操作

  POST /todos（Todo作成）：
    {"title": "牛乳を買う", "description": "スーパーで低脂肪乳を買う"}

  POST /todos（もう1つ作成）：
    {"title": "レポートを書く", "description": "金曜日までに提出"}

  GET /todos（一覧取得）：
    → 2件のTodoがJSON配列で返る

  PUT /todos/1（更新）：
    {"title": "牛乳を買う", "description": "スーパーで低脂肪乳を買う", "completed": true}

  DELETE /todos/2（削除）：
    → 削除成功のレスポンス

■ サーバー停止
  Ctrl + C


【ステップ3：pytest実行】

■ テスト実行
  uv run pytest -v

■ テスト結果の読み方
  PASSED  → テスト成功
  FAILED  → テスト失敗（失敗メッセージに原因が書かれている）
  最終行に「X passed」と表示 → 全テスト成功


【よくあるトラブルと対処法】

■ パターン1：import文のエラー（ModuleNotFoundError）
  原因：src/__init__.py の不足
  修正プロンプト：
    サーバー起動時に以下のエラーが出ます。修正してください。
    ModuleNotFoundError: No module named 'src'

■ パターン2：テストの失敗
  修正プロンプト：
    以下のテストが失敗しています。原因を特定して修正してください。
    （pytest の失敗ログをペースト）

  ※ 「テストを直して」と限定せず、APIとテストのどちらを直すべきかClaudeに判断させる

■ パターン3：ポート競合でサーバーが起動しない
  別ポートで起動：
    uv run uvicorn src.main:app --reload --port 8001

■ パターン4：構文エラー
  エラーメッセージをClaude Codeにそのまま貼る

■ Agent Teamsとの付き合い方
  - Agent Teamsの出力は「8割の完成品」と考える
  - 残り2割は通常のClaude Codeで仕上げる
  - エラーメッセージをコピーしてClaude Codeに貼るだけで修正できる

7.4 振り返り：1人 vs チーム
────────────────────────────────────────

■ サブエージェント vs Agent Teams — 3軸の比較表
  ┌──────────────────────┬──────────────────────────┬──────────────────────────────┐
  │ 比較項目             │ サブエージェント         │ Agent Teams                  │
  ├──────────────────────┼──────────────────────────┼──────────────────────────────┤
  │ 所要時間             │ タスクの合計（直列）     │ 最長タスク＋統合（並列）     │
  │ トークンコスト       │ メイン＋各サブの合計     │ メンバー数×セッション分      │
  │                      │                          │ （約3倍）                    │
  │ コンテキスト品質     │ ◎ 各サブが独立           │ ◎ 各メンバーが独立           │
  │ メンバー間連携       │ × できない               │ ○ メッセージング可           │
  │ セットアップ         │ 簡単（.mdファイルだけ）  │ tmux＋設定が必要             │
  │ 安定性               │ 高い                     │ 実験的（不安定な場合あり）   │
  │ 向いている規模       │ 小〜中規模               │ 中〜大規模                   │
  └──────────────────────┴──────────────────────────┴──────────────────────────────┘

■ Agent Teamsの強み
  ① 並列処理による時間短縮
  ② メンバー間のリアルタイムコミュニケーション
  ③ 各メンバーが専門領域に集中できる
  ④ 大規模タスクを1回の指示で完了できる

■ Agent Teamsの弱み
  ① トークンコストがメンバー数に比例して増える
  ② セットアップの手間（tmux、環境変数）
  ③ 実験的機能ゆえの不安定さ
  ④ セッション復元ができない（/resume 不可）
  ⑤ ファイル衝突のリスク管理が必要

■ 判断のポイント
  - 「並列化できるか？」が最初の判断基準
  - 順序依存のタスク → サブエージェントが適切
  - 独立して進められるタスク → Agent Teamsが速い
  → セクション8で判断フローチャートを学ぶ

■ 3つの武器のまとめ
  ① 通常のClaude Code   1人のAIに作業を頼む基本形
  ② サブエージェント     専門家を作って順番に仕事を任せる
  ③ Agent Teams          チームを組んで並列に開発する
